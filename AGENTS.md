# AGENTS.md

## 1. 项目/仓库用途简介

本仓库是一个基于 Streamlit 的科技资讯搜集与研判应用，同步承载以下主链路：

- 公司追踪：面向单个或多个科技公司并发抓取新闻，结合事件主档、时间线和上市公司金融补链输出战报。
- 行业扫描：按硬科技主题做每日宏观/行业早报，强调技术进展、产业链变化、市场催化与中国本地化落地。
- 文档导出：将分析结果导出为 PPT 和 Word，用于汇报、复盘和内部共享。

默认工作方式是：在 Git 同步仓库中修改代码，在带本地密钥的运行环境中做真实检查。

## 2. 关键目录说明

- `agent_app.py`：应用主入口，包含 Streamlit 页面、参数面板、任务编排、报告落盘和导出入口。
- `agents/`：分析与事件抽取层，主要负责事件主档、时间线和长文本分析。
- `tools/`：工具层，包含搜索、查询包、金融数据、PPT/Word 导出、记忆管理等模块。
- `.streamlit/secrets.toml.example`：本地密钥模板。真实密钥文件只应保存在本地，不应提交。
- `requirements.txt`：依赖清单。
- `template.pptx`：PPT 模板文件，导出布局依赖它，修改风险较高。
- `debug_local.py`：本地快速启动辅助脚本。

## 3. 常用命令

| 目的 | 命令 | 说明 |
| --- | --- | --- |
| 安装依赖 | `python -m pip install -r requirements.txt` | 首次拉起环境时使用 |
| 启动应用 | `python -m streamlit run agent_app.py` | 标准启动方式 |
| 本地调试启动 | `python debug_local.py` | 等价于用 Python 启动 Streamlit |
| 语法检查 | `Get-ChildItem agent_app.py, agents\*.py, tools\*.py \| ForEach-Object { python -m py_compile $_.FullName }` | 适合改动后做快速语法烟测 |
| 构建/打包 | 当前无独立 build 脚本 | PPT/Word 导出通过应用内流程触发 |
| Lint | 当前无仓库级 lint 配置 | 若未来新增 lint，需同步更新本文件 |

补充说明：

- 没有可用密钥时，应用可以启动，但搜索、模型调用和真实导出链路不能视为验证通过。
- 涉及真实验证时，优先在带 `.streamlit/secrets.toml` 的本地运行副本中执行。

## 4. 代码修改约束

- 默认最小改动，不做与当前问题无关的重构。
- 不提交真实密钥、账号信息、本地缓存、日志和临时产物。
- 修改 `agent_app.py` 时，尽量保持现有 `st.session_state` key、页面结构和导出字段兼容。
- 修改 `tools/company_query_packs.py` 或 `tools/intelligence_packs.py` 时，必须同时考虑：
  - 公司流与行业流是否仍然可用；
  - 中国专题过滤是否仍然成立；
  - 主题是否过度偏向单一新闻类型。
- 修改金融图表、PPT 版式或摘要分页逻辑时，必须联动检查导出结果，重点确认不重叠、不截断、不越界。
- 仓库可能长期处于脏工作树状态，不要覆盖、回退或格式化你未理解的现有改动。

## 5. 不建议修改的目录或文件

- `.venv/`
- `.idea/`
- `__pycache__/`
- `.streamlit/secrets.toml` 及任何真实密钥文件
- 生成产物与日志：`*.pptx`、`*.docx`、`*.png`、`*.json`、`*_stdout.log`、`*_stderr.log`
- `template.pptx`

说明：上述内容不是绝对禁止修改，而是除非任务明确要求，否则不要把它们作为默认改动对象。

## 6. 完成标准（Done when）

满足以下条件，才算本仓库中的一次修改真正完成：

- 目标问题已经在对应链路中处理，而不是只改了局部代码。
- 相关功能至少完成一次本地验证；若无法验证，原因已明确记录。
- 涉及检索、排序、主题包时，结果方向与任务目标一致，没有明显跑偏。
- 涉及图表、导出或版式时，输出文件可读、无明显重叠或截断。
- 未误提交真实密钥、临时文件或无关改动。
- 汇报中已经说明：改了什么、为什么改、如何验证、还有什么未验证。

## 7. 我作为初学者使用 Codex 时的注意事项

- 先把目标说清楚。不要只说“帮我看一下”，而要说明“要改什么、在哪个页面/文件出问题、希望改成什么样”。
- 如果你只想先看、不想立刻改代码，要直接写“先审视，再动手”。
- 如果存在两个目录，一个是 Git 同步仓库，一个是带本地密钥的运行副本，要把这件事先讲清楚。
- 如果你知道哪个文件大概率相关，直接点名文件；不知道也没关系，但要说明现象。
- 每次让 Codex 改完后，要求它回答四件事：改了什么、为什么、怎么验证、还有哪些没验证。
- 遇到高风险操作时，要明确要求它先停下来说明风险，再继续，例如：新增依赖、改模板、删文件、改核心配置。
